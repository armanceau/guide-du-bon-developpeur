# 🌿 Git & Versionning - TODO à faire

Git est l'outil de versioning le plus utilisé. Maîtriser Git est essentiel pour tout développeur.

## 🎯 Pourquoi utiliser Git ?

### Historique complet

- Savoir qui a fait quoi et quand
- Revenir en arrière si nécessaire
- Comprendre l'évolution du code

### Collaboration

- Travail en équipe facilité
- Branches pour isoler le travail
- Résolution de conflits

### Sauvegarde

- Code sauvegardé sur le serveur distant
- Protection contre la perte de données

## 📝 Commits

### Messages de commit

#### Format recommandé

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### Types de commits (Conventional Commits)

```bash
feat: Nouvelle fonctionnalité
fix: Correction de bug
docs: Documentation
style: Formatage (pas de changement de code)
refactor: Refactoring
test: Ajout/modification de tests
chore: Tâches de maintenance
perf: Amélioration de performance
ci: Intégration continue
build: Système de build
revert: Annulation d'un commit précédent
```

#### Exemples

```bash
✅ Bons commits

feat(auth): add password reset functionality

Implements password reset flow with email verification.
Users can now request a password reset link via email.

Closes #123

---

fix(api): handle null values in user endpoint

Previously crashed when user had no email.
Now returns default value.

---

docs: update installation instructions

Added steps for Windows users and troubleshooting section.

---

refactor(users): extract validation logic to separate service

No functional changes. Improves code organization and testability.
```

```bash
❌ Mauvais commits

"fixed stuff"
"update"
"wip"
"asdfasdf"
"changes"
"final version"
"final version 2"
"final final version"
```

### Bonnes pratiques de commits

#### 1. Commits atomiques

```bash
❌ Mauvais - Trop de changements dans un commit
git commit -m "Added login, fixed bugs, updated style"

✅ Bon - Un changement logique par commit
git commit -m "feat(auth): add login form"
git commit -m "fix(api): handle timeout in user fetch"
git commit -m "style(login): update button colors"
```

#### 2. Commits fréquents

```bash
✅ Commitez souvent
- Après chaque fonctionnalité complète
- Après chaque bug fix
- Avant de changer de contexte
- À la fin de la journée

❌ Évitez
- Gros commits avec plusieurs jours de travail
- "End of day" commits fourre-tout
```

#### 3. Testez avant de committer

```bash
# Workflow recommandé
1. Faire les changements
2. Tester localement
3. Relire les changements (git diff)
4. Stager les fichiers
5. Commit
6. Push

# Commandes
git status                    # Voir les changements
git diff                      # Voir les modifications
git add <fichiers>            # Stager
git commit -m "message"       # Commit
git push                      # Push vers remote
```

## 🌿 Branches

### Stratégies de branching

#### Git Flow

```
main (production)
  ↑
  develop (intégration)
    ↑
    ├─ feature/user-auth
    ├─ feature/payment
    ├─ bugfix/login-error
    └─ release/v1.2.0
```

```bash
# Branches principales
main          # Production, toujours stable
develop       # Développement, intégration

# Branches temporaires
feature/*     # Nouvelles fonctionnalités
bugfix/*      # Corrections de bugs
hotfix/*      # Corrections urgentes en production
release/*     # Préparation de release
```

#### GitHub Flow (Plus simple)

```
main (production)
  ↑
  ├─ feature/user-auth
  ├─ fix/button-style
  └─ docs/readme-update
```

```bash
# Une seule branche principale
main          # Production

# Branches de feature
feature/*     # Toutes les modifications
```

### Nommage des branches

```bash
✅ Bons noms
feature/user-authentication
feature/payment-integration
fix/login-button-not-working
hotfix/security-vulnerability
docs/api-documentation

❌ Mauvais noms
test
new-feature
fix
branch1
my-branch
```

### Workflow de branche

```bash
# 1. Créer une branche
git checkout -b feature/new-feature

# 2. Travailler et committer
git add .
git commit -m "feat: add new feature"

# 3. Pousser la branche
git push -u origin feature/new-feature

# 4. Créer une Pull Request sur GitHub/GitLab

# 5. Après merge, supprimer la branche
git checkout main
git pull
git branch -d feature/new-feature
```

## 🔧 Commandes essentielles

### Configuration initiale

```bash
# Identité
git config --global user.name "Votre Nom"
git config --global user.email "email@example.com"

# Éditeur
git config --global core.editor "code --wait"

# Branch par défaut
git config --global init.defaultBranch main

# Couleurs
git config --global color.ui auto
```

### Commandes de base

```bash
# Initialiser un repo
git init

# Cloner un repo
git clone <url>

# Statut
git status

# Voir les différences
git diff                    # Unstaged changes
git diff --staged           # Staged changes
git diff main feature       # Entre deux branches

# Stager des fichiers
git add <file>              # Un fichier
git add .                   # Tous les fichiers
git add *.js                # Pattern
git add -p                  # Interactif

# Commit
git commit -m "message"
git commit -am "message"    # Add + commit (fichiers trackés)
git commit --amend          # Modifier dernier commit

# Historique
git log
git log --oneline
git log --graph --all
git log --author="John"
git log --since="2 weeks ago"

# Branches
git branch                  # Lister les branches
git branch <name>           # Créer une branche
git checkout <branch>       # Changer de branche
git checkout -b <branch>    # Créer et changer
git branch -d <branch>      # Supprimer (merged)
git branch -D <branch>      # Forcer suppression

# Merge
git merge <branch>

# Remote
git remote -v               # Voir les remotes
git remote add origin <url> # Ajouter remote
git fetch                   # Récupérer changes
git pull                    # Fetch + merge
git push                    # Pousser changes
git push -u origin main     # Première fois
```

### Commandes avancées

```bash
# Rebase (réécrire l'historique)
git rebase main             # Rebaser sur main
git rebase -i HEAD~3        # Rebase interactif (3 derniers commits)

# Stash (mettre de côté)
git stash                   # Sauvegarder modifications
git stash list              # Lister les stashes
git stash apply             # Réappliquer
git stash pop               # Appliquer et supprimer
git stash drop              # Supprimer

# Reset
git reset --soft HEAD~1     # Annuler commit, garder changes staged
git reset --mixed HEAD~1    # Annuler commit et staging
git reset --hard HEAD~1     # Annuler commit et changes (⚠️ DANGER)

# Revert (annuler avec nouveau commit)
git revert <commit>         # Plus sûr que reset

# Cherry-pick
git cherry-pick <commit>    # Appliquer un commit spécifique

# Blame
git blame <file>            # Voir qui a modifié quoi

# Bisect (trouver le commit qui a introduit un bug)
git bisect start
git bisect bad              # Ce commit est mauvais
git bisect good <commit>    # Ce commit était bon
# Git testera automatiquement les commits entre les deux
```

## 🔥 Résolution de conflits

### Workflow de résolution

```bash
# 1. Un conflit apparaît lors d'un merge
git merge feature-branch
# CONFLICT (content): Merge conflict in file.js

# 2. Voir les fichiers en conflit
git status

# 3. Ouvrir le fichier et résoudre
<<<<<<< HEAD
const version = 1;
=======
const version = 2;
>>>>>>> feature-branch

# Choisir la bonne version ou combiner
const version = 2;

# 4. Marquer comme résolu
git add file.js

# 5. Continuer le merge
git commit
```

### Stratégies de merge

```bash
# Merge standard (crée un commit de merge)
git merge feature-branch

# Squash (combine tous les commits en un)
git merge --squash feature-branch

# No fast-forward (toujours créer un commit de merge)
git merge --no-ff feature-branch

# Rebase avant merge (historique linéaire)
git checkout feature-branch
git rebase main
git checkout main
git merge feature-branch
```

## ✅ Bonnes pratiques

### 1. Synchronisez souvent

```bash
# Avant de commencer à travailler
git pull

# Plusieurs fois par jour
git fetch
git status
```

### 2. Branches courtes

```bash
✅ Branches de 1-3 jours max
❌ Branches qui vivent pendant des semaines
```

### 3. Revoyez avant de pousser

```bash
# Voir ce que vous allez pousser
git log origin/main..HEAD
git diff origin/main..HEAD
```

### 4. Utilisez des alias

```bash
# Dans ~/.gitconfig
[alias]
  st = status
  co = checkout
  br = branch
  ci = commit
  last = log -1 HEAD
  unstage = reset HEAD --
  visual = log --graph --oneline --all
  amend = commit --amend --no-edit

# Utilisation
git st
git visual
```

## 🚨 Erreurs courantes et solutions

### J'ai commité sur la mauvaise branche

```bash
# Solution: cherry-pick sur la bonne branche
git checkout correct-branch
git cherry-pick <commit-hash>
git checkout wrong-branch
git reset --hard HEAD~1
```

### J'ai besoin d'annuler mon dernier commit

```bash
# Garder les modifications
git reset --soft HEAD~1

# Supprimer les modifications (⚠️ DANGER)
git reset --hard HEAD~1
```

### J'ai commité des secrets (API keys, passwords)

```bash
# ⚠️ ATTENTION: Changer immédiatement les secrets!

# Supprimer du dernier commit
git reset --soft HEAD~1
# Éditer le fichier et refaire le commit

# Si déjà pushé
# Utiliser git filter-branch ou BFG Repo-Cleaner
# Puis force push (coordonner avec l'équipe!)
```

### Mon historique est sale

```bash
# Nettoyer les derniers commits avec rebase interactif
git rebase -i HEAD~5

# Options:
# pick = garder le commit
# reword = changer le message
# squash = fusionner avec le précédent
# drop = supprimer le commit
```
