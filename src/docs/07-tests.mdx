# Tests - TODO à faire

Les tests sont essentiels pour garantir la qualité, la fiabilité et la maintenabilité de votre code.

## Pourquoi tester ?

### Confiance

- Déployer sans peur
- Refactorer en toute sécurité
- Modifier le code existant sans casser

### Documentation

- Les tests servent de documentation vivante
- Montrent comment utiliser le code
- Explicitent les comportements attendus

### Qualité

- Détection précoce des bugs
- Force à écrire du code testable (donc mieux conçu)
- Prévention des régressions

### Tests unitaires (80%)

Les tests unitaires sont très nombreux, rapides et peu coûteux.

Testent une unité de code isolée (fonction, méthode, composant)

### Tests d'intégration (15%)

Ils sont moyennement nombreux.

Testent l'interaction entre plusieurs unités

### Tests E2E (5%)

Les tests end-to-end sont peu nombreux, lents et très coûteux.

Testent l'application complète du point de vue utilisateur

## Tests unitaires

### Principes

#### 1. GVT Pattern : Given, When, Then

Ce pattern est utilisé dans le BDD (Behavior-Driven Development) pour définir clairement les cas testés ainsi que les critères d'acceptances.

Il est aussi utilisé pour améliorer la compréhension du test. Cette pratique est plutôt efficace pour la compréhension des juniors.

Exemple :

```typescript
@Test
void additionneDeuxNombres() {
    // Given
    int a = 5;
    int b = 3;
    Calculator calculator = new Calculator();

    // When
    int result = calculator.add(a, b);

    // Then
    assertEquals(8, result);
}
```

#### 2. `@DisplayName`

L’annotation JUnit `@DisplayName` permet de donner un nom lisible et explicite à un test.

Elle est utilisée pour améliorer la compréhension des tests, notamment dans les rapports d’exécution.

Pourquoi l’utiliser ?

- Rendre les tests compréhensibles par tous (même non développeurs)
- Décrire clairement le comportement testé
- Améliorer la lisibilité dans les outils de CI / rapports

Exemple :

```typescript
@Test
@DisplayName("Addition de deux nombres retourne le bon résultat")
void additionneDeuxNombres() {
    // Given
    int a = 5;
    int b = 3;
    Calculator calculator = new Calculator();

    // When
    int result = calculator.add(a, b);

    // Then
    assertEquals(8, result);
}
```

### Bonnes pratiques

#### 1. Un test = un concept

Test incorrect :

```typescript
@Test
void userOperations() {
    assertTrue(createUser());
    assertTrue(deleteUser());
    assertTrue(updateUser());
}
```

Test correct :

```typescript
@Test
void createsANewUser() {
    assertTrue(createUser());
}

@Test
void deletesAnExistingUser() {
    assertTrue(deleteUser());
}
```

#### 2. Noms de tests descriptifs

Test incorrect :

```typescript
@Test
void test1() {
    // ...
}

@Test
void itWorks() {
    // ...
}
```

Test correct :

```typescript
@Test
void returnsNullWhenUserNotFound() {
    // ...
}

@Test
void throwsErrorWhenEmailInvalid() {
    // ...
}
```

#### 3. Tests indépendants

Tests incorrects (tests dépendants) :

```typescript
User user;

@Test
void createsUser() {
    user = createUser();
    assertNotNull(user);
}

@Test
void updatesUser() {
    user.setName("New Name");
    assertEquals("New Name", user.getName());
}
```

Tests corrects :

```typescript
@Test
void createsUser() {
    User user = createUser();
    assertNotNull(user);
}

@Test
void updatesUserName() {
    User user = createUser(); // Crée son propre user
    user.setName("New Name");
    assertEquals("New Name", user.getName());
}
```

## Tests d'intégration

Testent plusieurs unités qui travaillent ensemble.

```typescript
class UserApiIntegrationTest {

    static TestDatabase database;

    @BeforeAll
    static void setup() {
        // Setup real database for integration tests
        database = TestDatabase.setupTestDatabase();

        // Configure l'URL de base de l'API
        RestAssured.baseURI = "http://localhost:8080";
    }

    @AfterAll
    static void teardown() {
        database.close();
    }

    @Test
    void postUsersCreatesANewUser() {
        // Given
        String requestBody = """
            {
                "email": "test@example.com",
                "password": "password123"
            }
        """;

        // When
        Response response = RestAssured
                .given()
                .contentType("application/json")
                .body(requestBody)
                .when()
                .post("/users");

        // Then
        assertEquals(201, response.getStatusCode());

        String userId = response.jsonPath().getString("id");
        String email = response.jsonPath().getString("email");
        User user = database.findById(userId);

        assertNotNull(userId);
        assertEquals("test@example.com", email);
        assertNotNull(user);
    }
}
```

## Tests End-to-End (E2E)

Testent l'application complète dans un navigateur réel.

```typescript
describe("User authentication", () => {
  it("should redirect to dashboard after successful login", () => {
    cy.visit("/login");

    cy.findByRole("textbox", { name: /email/i }).type("test@example.com");
    cy.findByRole("textbox", { name: /password/i }).type("password123");
    cy.findByRole("button", { name: /login/i }).click();

    cy.url().should("include", "/dashboard");

    cy.findByRole("heading", { name: /welcome/i }).should("be.visible");
  });
});
```

Privilégier `findByRole` :

La fonction `findByRole` permet de cibler les éléments comme le ferait un utilisateur (via leur rôle et leur texte visible), ce qui rend les tests plus lisibles et réalistes.

Il est aussi plus robuste que `cy.get()` car moins dépendant de la structure HTML.

Enfin, il encourage une meilleure accessibilité en s’appuyant sur les standards ARIA.

## Tests unitaires (frontend - React)

Voici quelques bonnes pratiques pour les tests unitaires front :
- Tester le comportement, pas les détails techniques internes
- Éviter de tester les classes CSS sauf si elles ont un impact fonctionnel
- Garder des tests simples et ciblés

Exemple :
```typescript
import { render, screen, fireEvent } from "@testing-library/react";
import { Button } from "./Button";

describe("Button Component", () => {
  test("renders button with label", () => {
    render(<Button label="Click me" />);
    expect(screen.getByText("Click me")).toBeInTheDocument();
  });

  test("calls onClick when clicked", () => {
    const handleClick = jest.fn();
    render(<Button label="Click me" onClick={handleClick} />);

    fireEvent.click(screen.getByText("Click me"));

    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  test("is disabled when disabled prop is true", () => {
    render(<Button label="Click me" disabled />);
    expect(screen.getByText("Click me")).toBeDisabled();
  });

  test("applies correct variant class", () => {
    render(<Button label="Click me" variant="primary" />);
    const button = screen.getByText("Click me");
    expect(button).toHaveClass("btn--primary");
  });
});
```

### Objectifs raisonnables

- **Statements** : 80%+
- **Branches** : 80%+
- **Functions** : 80%+
- **Lines** : 80%+

L'objectif est d'obtenir une couverture d'au moins 80% sur l'ensemble de l'application (front/back).

## TDD (Test-Driven Development)

### Le cycle Red-Green-Refactor

1. 🔴 RED : Écrire un test qui échoue
2. 🟢 GREEN : Écrire le minimum de code pour passer le test
3. 🔵 REFACTOR : Améliorer le code sans changer le comportement
4. Répéter

### Exemple TDD

```typescript
// 1. 🔴 RED - Test qui échoue
test("calculates average of array", () => {
  expect(average([1, 2, 3])).toBe(2);
});

// 2. 🟢 GREEN - Implémentation minimale
function average(numbers) {
  return numbers.reduce((a, b) => a + b) / numbers.length;
}

// 3. 🔵 REFACTOR - Améliorer (gérer edge cases)
function average(numbers) {
  if (!numbers || numbers.length === 0) {
    return 0;
  }
  return numbers.reduce((a, b) => a + b, 0) / numbers.length;
}

// Ajouter test pour le edge case
test("returns 0 for empty array", () => {
  expect(average([])).toBe(0);
});
```
