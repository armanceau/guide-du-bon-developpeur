# Code Reviews

Les revues de code constituent un processus essentiel pour garantir la qualité du code et favoriser le partage
de connaissances au sein d’une équipe.

Bien qu’elles puissent parfois paraître longues ou fastidieuses, elles
jouent un rôle clé dans l’amélioration continue du code et des pratiques de développement.

## Pourquoi faire des code reviews ?

### Amélioration de la qualité

- Détection précoce des bugs
- Identification des problèmes de performance
- Respect des standards de code
- Amélioration de la maintenabilité

### Partage de connaissances

- Transfert de compétences entre développeurs
- Familiarisation avec différentes parties du codebase
- Apprentissage de nouvelles techniques
- Meilleur compéhension du métier

### Cohésion d'équipe

- Responsabilité collective du code
- Communication améliorée
- Culture d'amélioration continue

## Bonnes pratiques pour soumettre du code

### 1. Gardez les MR/PR petites et ciblées

```
❌ PR avec 50 fichiers modifiés, 2000 lignes changées
✅ PR avec 5-10 fichiers, 200-300 lignes max
```

### 2. Rédigez une description claire

Chaque PR devrait contenir :

- **Nom de la branche** : Utiliser une convention claire, par exemple : `feat/Nom_ticket_courte_description_en_5_mots`
- **Titre de la PR** : Une phrase courte qui résume le changement
- **Contexte** : Un résumé rapide du ticket ou du problème traité
- **Description** : Pourquoi ce changement ?
- **Changements** : Qu'est-ce qui a été modifié ?
- **Tests** : Comment vérifier que ça fonctionne ?
- **Screenshots** : Si pertinent (UI)

### 3. Auto-review avant de soumettre

- Relisez votre propre code
- Testez localement
- Vérifiez que les tests passent
- Supprimez les console.log et code commenté

## Bonnes pratiques pour relire du code

Soyez constructif et bienveillant :

```
❌ "Ce code est nul"
✅ "Cette approche pourrait être simplifiée. Que penses-tu de...?"

❌ "Tu ne sais pas coder ?"
✅ "J'ai une suggestion qui pourrait améliorer la lisibilité..."
```

## Etapes de la review

- Regarder l’architecture : les fichiers sont-ils au bon endroit ? La structure respecte-t-elle l’architecture du projet ?
- Lire le ticket : comprendre le besoin fonctionnel et les attentes décrites.
- Lire les tests : vérifier que le comportement attendu est bien couvert et comprendre la logique métier.
- Lire l’implémentation : le code répond-il correctement au besoin métier ?
- Vérifier la lisibilité : les noms de fonctions, variables et composants sont-ils explicites ?
- Évaluer la simplicité : le code est-il clair et maintenable ou pourrait-il être simplifié ?
- Vérifier les cas limites : les erreurs, cas particuliers ou états inattendus sont-ils bien gérés ?
- Contrôler la cohérence : le code respecte-t-il les conventions et les pratiques du projet ?

## Timing des reviews

- Objectif : reviewer dans les **2-4 heures**
- Bloquez du temps dédié aux reviews
- Options : - Le matin : avant de démarrer ses tâches de la journée (préférence) - Avant d'aller manger pour finir sa matinée - Après avoir finit la pause déjeuner avant de se remettre dans son travail - En fin de journée
