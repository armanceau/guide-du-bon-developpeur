# 🔍 Code Reviews

Les revues de code sont un processus essentiel pour maintenir la qualité du code et favoriser le partage de connaissances au sein d'une équipe.

## 🎯 Pourquoi faire des code reviews ?

### Amélioration de la qualité

- Détection précoce des bugs
- Identification des problèmes de performance
- Respect des standards de code
- Amélioration de la maintenabilité

### Partage de connaissances

- Transfert de compétences entre développeurs
- Familiarisation avec différentes parties du codebase
- Apprentissage de nouvelles techniques

### Cohésion d'équipe

- Responsabilité collective du code
- Communication améliorée
- Culture d'amélioration continue

## ✅ Bonnes pratiques pour soumettre du code

### 1. Gardez les PR petites et ciblées

```
❌ Mauvais : PR avec 50 fichiers modifiés, 2000 lignes changées
✅ Bon : PR avec 5-10 fichiers, 200-300 lignes max
```

### 2. Rédigez une description claire

Chaque PR devrait contenir :

- **Contexte** : Pourquoi ce changement ?
- **Changements** : Qu'est-ce qui a été modifié ?
- **Tests** : Comment vérifier que ça fonctionne ?
- **Screenshots** : Si pertinent (UI)

### 3. Auto-review avant de soumettre

- Relisez votre propre code
- Testez localement
- Vérifiez que les tests passent
- Supprimez les console.log et code commenté

### 4. Fournissez du contexte

```markdown
## Problème

L'utilisateur ne pouvait pas se déconnecter sur mobile

## Solution

Ajout d'un bouton de déconnexion dans le menu mobile

## Tests

- ✅ Test unitaire du bouton
- ✅ Test E2E du flux de déconnexion
- ✅ Testé sur iOS et Android
```

## 🔎 Bonnes pratiques pour reviewer

### 1. Soyez constructif et bienveillant

```
❌ "Ce code est nul"
✅ "Cette approche pourrait être simplifiée. Que penses-tu de...?"

❌ "Tu ne sais pas coder ?"
✅ "J'ai une suggestion qui pourrait améliorer la lisibilité..."
```

### 2. Distinguez les niveaux de criticité

```
🔴 BLOCKING : Problème critique (bug, faille de sécurité)
🟡 SUGGESTION : Amélioration optionnelle
💡 QUESTION : Demande de clarification
✨ COMPLIMENT : Soulignez le bon code !
```

### 3. Concentrez-vous sur le code, pas sur la personne

```
❌ "Tu as fait une erreur stupide"
✅ "Cette logique pourrait causer un bug si..."
```

### 4. Expliquez le "pourquoi"

```
❌ "Change ça"
✅ "Cette approche pourrait causer des problèmes de mémoire
    car elle crée une nouvelle instance à chaque render.
    Considère utiliser useMemo ici."
```

### 5. Proposez des solutions

```
❌ "C'est pas bien"
✅ "Cette approche fonctionne, mais on pourrait simplifier avec :
    [code example]"
```

## ⏱️ Timing des reviews

### Réactivité

- Objectif : reviewer dans les **2-4 heures**
- Maximum acceptable : **24 heures**
- Bloquez du temps dédié aux reviews

### Ordre de priorité

1. Hotfixes (immédiat)
2. PR bloquantes
3. PR normales
4. Refactoring / non urgent

## 📋 Checklist du reviewer

### Fonctionnalité

- [ ] Le code fait ce qu'il est censé faire
- [ ] Les edge cases sont gérés
- [ ] Pas de régression introduite

### Qualité du code

- [ ] Code lisible et compréhensible
- [ ] Noms de variables/fonctions explicites
- [ ] Pas de duplication inutile
- [ ] Respect des conventions du projet

### Tests

- [ ] Tests présents et pertinents
- [ ] Couverture suffisante
- [ ] Tests passent

### Performance

- [ ] Pas de problèmes de performance évidents
- [ ] Requêtes optimisées
- [ ] Pas de memory leaks

### Sécurité

- [ ] Pas de failles de sécurité
- [ ] Validation des inputs
- [ ] Données sensibles protégées

## 💬 Templates de commentaires

### Demander une clarification

```
💡 Peux-tu expliquer pourquoi tu as choisi cette approche ?
Je me demandais si [alternative] pourrait aussi fonctionner ?
```

### Suggérer une amélioration

```
🟡 SUGGESTION : Cette fonction est un peu longue.
Que penses-tu de l'extraire en plusieurs fonctions plus petites ?
```

### Signaler un problème

```
🔴 BLOCKING : Cette condition ne gère pas le cas où user
est null, ce qui pourrait crasher l'app.
```

### Donner un compliment

```
✨ Super idée d'utiliser cette pattern ! C'est beaucoup plus
lisible qu'avant.
```
