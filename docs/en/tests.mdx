# Tests

Tests are essential to guarantee the quality, reliability, and maintainability of your code.

## Why test?

### Confidence

- Deploy without fear
- Refactor safely
- Modify existing code without breaking things

### Documentation

- Tests serve as living documentation
- Show how to use the code
- Make expected behaviors explicit

### Quality

- Early bug detection
- Forces you to write testable code (and therefore better designed)
- Regression prevention

## The test pyramid

The test pyramid (Google's version of Mike Cohn's model) recommends a majority of unit tests, an intermediate integration layer, and few E2E tests.

Reference: [Software Engineering at Google](https://abseil.io/resources/swe-book/html/ch11.html)

### Maintenance cost

- Unit tests: a few minutes
- Integration tests: a few hours to half a day
- E2E tests: half a day to several days

<Callout type="info">
  The closer a test is to the user (E2E), the more expensive it is to maintain.

  Conversely, the more isolated it is (unit), the faster and more stable it is.
</Callout>

### Unit tests (80%)

Unit tests are numerous, fast, and inexpensive.

They test an isolated unit of code (function, method, component).

### Integration tests (15%)

There are a moderate number of them.

They test the interaction between multiple units.

### E2E tests (5%)

End-to-end tests are few in number, slow, and very expensive.

They test the full application from the user's perspective.

## Beyoncé Rule

- If your code really matters, put a test on it
- This rule applies at any scale
- An important behavior without a test is immediate debt

## Fix It, Test It Rule

- Bug found = test required
- No fix without an associated test
- Every bug becomes a permanently covered case

## Unit tests

### Principles

#### 1. GWT Pattern: Given, When, Then

This pattern is used in BDD (Behavior-Driven Development) to clearly define the tested cases and acceptance criteria.

It is also used to improve test comprehension. This practice is particularly effective for junior developers.

Example:

```typescript
@Test
void addsTwoNumbers() {
    // Given
    int a = 5;
    int b = 3;
    Calculator calculator = new Calculator();

    // When
    int result = calculator.add(a, b);

    // Then
    assertEquals(8, result);
}
```

#### 2. `@DisplayName`

The JUnit `@DisplayName` annotation gives a readable and explicit name to a test.

It is used to improve test comprehension, especially in execution reports.

Why use it?

- Make tests understandable by everyone (even non-developers)
- Clearly describe the tested behavior
- Improve readability in CI tools / reports

Example:

```typescript
@Test
@DisplayName("Adding two numbers returns the correct result")
void addsTwoNumbers() {
    // Given
    int a = 5;
    int b = 3;
    Calculator calculator = new Calculator();

    // When
    int result = calculator.add(a, b);

    // Then
    assertEquals(8, result);
}
```

### Best practices

#### 1. One test = one concept

Incorrect test:

```typescript
@Test
void userOperations() {
    assertTrue(createUser());
    assertTrue(deleteUser());
    assertTrue(updateUser());
}
```

Correct test:

```typescript
@Test
void createsANewUser() {
    assertTrue(createUser());
}

@Test
void deletesAnExistingUser() {
    assertTrue(deleteUser());
}
```

#### 2. Descriptive test names

Incorrect test:

```typescript
@Test
void test1() {
    // ...
}

@Test
void itWorks() {
    // ...
}
```

Correct test:

```typescript
@Test
void returnsNullWhenUserNotFound() {
    // ...
}

@Test
void throwsErrorWhenEmailInvalid() {
    // ...
}
```

#### 3. Independent tests

Incorrect tests (dependent tests):

```typescript
User user;

@Test
void createsUser() {
    user = createUser();
    assertNotNull(user);
}

@Test
void updatesUser() {
    user.setName("New Name");
    assertEquals("New Name", user.getName());
}
```

Correct tests:

```typescript
@Test
void createsUser() {
    User user = createUser();
    assertNotNull(user);
}

@Test
void updatesUserName() {
    User user = createUser(); // Creates its own user
    user.setName("New Name");
    assertEquals("New Name", user.getName());
}
```

## Integration tests

Test multiple units working together.

```typescript
class UserApiIntegrationTest {

    static TestDatabase database;

    @BeforeAll
    static void setup() {
        // Setup real database for integration tests
        database = TestDatabase.setupTestDatabase();

        // Configure the API base URL
        RestAssured.baseURI = "http://localhost:8080";
    }

    @AfterAll
    static void teardown() {
        database.close();
    }

    @Test
    void postUsersCreatesANewUser() {
        // Given
        String requestBody = """
            {
                "email": "test@example.com",
                "password": "password123"
            }
        """;

        // When
        Response response = RestAssured
                .given()
                .contentType("application/json")
                .body(requestBody)
                .when()
                .post("/users");

        // Then
        assertEquals(201, response.getStatusCode());

        String userId = response.jsonPath().getString("id");
        String email = response.jsonPath().getString("email");
        User user = database.findById(userId);

        assertNotNull(userId);
        assertEquals("test@example.com", email);
        assertNotNull(user);
    }
}
```

## End-to-End (E2E) tests

Test the full application in a real browser.

```typescript
describe("User authentication", () => {
  it("should redirect to dashboard after successful login", () => {
    cy.visit("/login");

    cy.findByRole("textbox", { name: /email/i }).type("test@example.com");
    cy.findByRole("textbox", { name: /password/i }).type("password123");
    cy.findByRole("button", { name: /login/i }).click();

    cy.url().should("include", "/dashboard");

    cy.findByRole("heading", { name: /welcome/i }).should("be.visible");
  });
});
```

Prefer `findByRole`:

The `findByRole` function targets elements the way a user would (via their role and visible text), making tests more readable and realistic.

It is also more robust than `cy.get()` because it is less dependent on HTML structure.

Finally, it encourages better accessibility by relying on ARIA standards.

## Unit tests (frontend - React)

Some best practices for frontend unit tests:

- Test behavior, not internal technical details
- Avoid testing CSS classes unless they have a functional impact
- Keep tests simple and targeted

Example:

```typescript
import { render, screen, fireEvent } from "@testing-library/react";
import { Button } from "./Button";

describe("Button Component", () => {
  test("renders button with label", () => {
    render(<Button label="Click me" />);
    expect(screen.getByText("Click me")).toBeInTheDocument();
  });

  test("calls onClick when clicked", () => {
    const handleClick = jest.fn();
    render(<Button label="Click me" onClick={handleClick} />);

    fireEvent.click(screen.getByText("Click me"));

    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  test("is disabled when disabled prop is true", () => {
    render(<Button label="Click me" disabled />);
    expect(screen.getByText("Click me")).toBeDisabled();
  });

  test("applies correct variant class", () => {
    render(<Button label="Click me" variant="primary" />);
    const button = screen.getByText("Click me");
    expect(button).toHaveClass("btn--primary");
  });
});
```

## Code coverage

### Reasonable targets

- **Statements**: 80%+
- **Branches**: 80%+
- **Functions**: 80%+
- **Lines**: 80%+

The goal is to achieve at least 80% coverage across the entire application (front/back).

<Callout type="warning">
  100% coverage does not guarantee the absence of bugs!

  50% of relevant tests is better than 100% of useless tests.
</Callout>

## Checklist for good tests

### Structure

- [ ] Tests organized with describe/test
- [ ] Descriptive and clear names (`@DisplayName`)
- [ ] GWT pattern applied
- [ ] Independent tests

### Quality

- [ ] Test behavior, not implementation
- [ ] Cover normal cases and edge cases
- [ ] Use appropriate mocks (avoid as much as possible — we want to test the real behavior of the application)
- [ ] Fast to execute

### Maintenance

- [ ] Easy to understand
- [ ] Easy to modify

## TDD (Test-Driven Development)

### The Red-Green-Refactor cycle

1. RED: Write a failing test
2. GREEN: Write the minimum code to pass the test
3. REFACTOR: Improve the code without changing the behavior
4. Repeat

### TDD example

```typescript
// 1. RED - Failing test
test("calculates average of array", () => {
  expect(average([1, 2, 3])).toBe(2);
});

// 2. GREEN - Minimal implementation
function average(numbers) {
  return numbers.reduce((a, b) => a + b) / numbers.length;
}

// 3. REFACTOR - Improve (handle edge cases)
function average(numbers) {
  if (!numbers || numbers.length === 0) {
    return 0;
  }
  return numbers.reduce((a, b) => a + b, 0) / numbers.length;
}

// Add test for the edge case
test("returns 0 for empty array", () => {
  expect(average([])).toBe(0);
});
```
