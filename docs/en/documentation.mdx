# Documentation

Good documentation makes a project easier to use, maintain, and hand off.

## Why document?

### For others

- Facilitates onboarding
- Makes code more understandable
- Reduces informal support requests

### For yourself

- You will forget how the code works
- Documentation helps clarify ideas
- It prevents rediscovering the same business rules

### For the project

- Reduces dependency on individuals
- Facilitates maintenance
- Improves overall quality

## Types of documentation

### README

The README should allow anyone to start the project without having to guess the steps.

### API documentation

- JSDoc for JavaScript/TypeScript projects
- Annotations and examples for REST APIs
- Swagger or OpenAPI to clearly expose contracts

### Code documentation

- Types already help document
- Comments should explain the *why*, not repeat the code
- Explicit code is better than a long perishable comment

## Best practices

- Keep examples short and up to date
- Document team conventions
- Update docs at the same time as the code
- Remove documentation that has become obsolete

## What to avoid

- Documenting what is already obvious in the code
- Leaving outdated docs for more than a few sprints
- Replacing poor code structure with comments
